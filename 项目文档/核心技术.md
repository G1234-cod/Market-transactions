## 一、FastAPI 异步 Web 框架

### 1.1 技术原理

**什么是异步编程？**

传统的 Web 框架（如 Flask）采用同步模型：一个请求进来，服务器分配一个线程从头到尾处理，直到返回响应。如果处理过程中需要等待外部服务（比如调用 LLM API 等 3 秒），这个线程就**空转等待**，无法处理其他请求。这意味着同时只能服务有限数量的用户。

异步编程的核心是**事件循环**。当程序遇到需要等待的操作时（如网络请求、数据库查询），它会把这个任务"挂起"，转而去处理其他已经就绪的任务。等等待的操作完成后，再回来继续执行。这样**一个线程就可以同时处理成千上万个请求**，让等待时间被充分利用。

**ASGI 是什么？**

ASGI 是 Python 异步 Web 服务器网关接口，是 WSGI 的升级版。WSGI 是同步的，一次只能处理一个请求；ASGI 支持异步，允许多个请求并发处理，还支持 WebSocket 长连接和流式响应（SSE）。

**FastAPI 的定位**

FastAPI 是目前 Python 最流行的异步 Web 框架，基于 Starlette（ASGI 框架）和 Pydantic（数据校验）。它既提供了异步能力，又自带自动生成 API 文档的功能，同时通过类型注解提供编辑器智能提示。

### 1.2 项目实现

**为什么选 FastAPI？**

我们的系统有大量等待场景：调用 DeepSeek API 需要 2-5 秒、查询 MySQL 数据库、读写图片文件。这些都是 IO 操作，异步可以让单个服务器同时处理更多请求。FastAPI 原生支持异步，正好匹配这个需求。

FastAPI 还提供了依赖注入系统，可以方便地抽离通用逻辑——比如 JWT 认证、速率限制，这些都可以写成"依赖项"，路由函数只需声明需要什么，框架自动注入。

**异步数据库驱动**

项目中使用了 `aiomysql` 作为 MySQL 驱动，这是异步版本的 PyMySQL。两者的区别在于：同步驱动执行查询时会阻塞线程，异步驱动则会在等待数据库返回时让出控制权，让事件循环处理其他任务。

同时使用 `aiofiles` 处理文件读写，同样避免阻塞。整个请求链路——从接收请求、调用 LLM、读写数据库、返回响应——全部是非阻塞的。

## 二、LLM 多模型协作体系

### 2.1 技术原理

**大语言模型怎么工作？**

大语言模型本质是**下一个词预测**。给模型一段文本，它会计算下一个词最可能是什么，然后输出。输出的词又作为输入，继续预测下一个词，如此往复生成完整文本。

这个过程是**自回归**的——每次生成都依赖之前生成的内容。这就是为什么生成需要时间：一篇 200 字的文案，模型要跑 200 次预测。

**Token 是什么？**

Token 是模型处理文本的最小单位。中文一个汉字通常占 1-2 个 token，英文一个单词占 1 个 token。模型按 token 收费，DeepSeek 生成一篇 200 字的文案约消耗 300-500 tokens。

**多模态模型**

普通 LLM 只能处理文本。多模态模型（如 Qwen-VL）额外增加了**视觉编码器**，能将图片转换成模型能理解的"视觉 token"。原理是把图片切成小块，每块转换成特征向量，然后和文本一起送入语言模型处理。这让模型既能"看"图片，又能"说"文字。

**为什么大模型更准？**

模型参数量越大，能学习的模式越复杂。Qwen-VL-Max 有千亿参数，经过海量图文数据训练，视觉理解能力远超几十 MB 的本地模型。但参数多也意味着：调用成本高、响应慢、需要专用 GPU 显存。

### 2.2 项目实现

**为什么不用一个模型解决全部？**

GPT-4V 虽然既能看图又能写文案，但有三方面问题：第一是成本，识别一次加上生成一次的费用远超组合方案；第二是速度，通用大模型做专项任务不如专用模型快；第三是输出稳定性，难以保证每次都输出可解析的结构化数据。

因此采用"各取所长"的策略：让 Qwen-VL-Max 专注做视觉识别，让 DeepSeek 专注做中文文案生成和定价推理。

**策略模式设计**

系统将所有 LLM 客户端设计为统一的抽象接口，包含"普通对话"和"流式对话"两个方法。DeepSeek 客户端、Qwen-VL 客户端、定价客户端各自实现这个接口。

这样做的好处是：如果以后要换模型，只需新增一个客户端类，业务代码完全不用改动。这种"面向接口编程"的方式提升了系统的可扩展性。

**Qwen-VL 双通道容错**

Qwen-VL 有两种调用方式：一是 OpenAI 兼容接口，二是阿里云 DashScope 原生 SDK。两种方式的认证、请求格式、响应解析都不同。

系统实现了一个容错机制：优先使用 OpenAI 兼容接口（更标准），如果失败则自动切换到 DashScope 原生 SDK。两个通道都失败才会报错。这大大提升了视觉识别功能的稳定性。

**审计日志**

每次调用 LLM，系统自动记录：哪个用户调用的、调用了哪个模型、用了多长时间、成功还是失败、输入输出的摘要（截断保护隐私）。这些数据有三个用途：成本核算（按用户统计用量）、问题追溯（用户反馈时查原始记录）、质量监控（跟踪成功率变化趋势）。

## 三、SSE 流式文案生成

### 3.1 技术原理

**什么是 SSE？**

SSE（Server-Sent Events，服务端推送事件）是 HTML5 标准的一部分，允许服务器通过 HTTP 连接向客户端持续推送数据。客户端使用 `EventSource` API 或 `fetch` 接收。

**SSE vs WebSocket**

两者都支持服务端向客户端推送数据，但定位不同：

| | SSE | WebSocket |
|---|---|---|
| 通信方向 | 单向（服务器→客户端） | 双向（客户端↔服务器） |
| 协议 | 标准 HTTP | 需要从 HTTP 升级 |
| 自动重连 | 浏览器内置 | 需要手动实现 |
| 适用场景 | 通知推送、流式输出 | 实时聊天、协作编辑 |

文案生成场景中，客户端已经在 POST 请求里把所有参数传给了服务器，后续只需要服务器把生成的内容推回来，不需要双向通信。SSE 最合适。

**为什么流式优于一次性返回？**

LLM 生成完整文案需要 10-20 秒。如果用普通接口，用户要盯着空白页面等十几秒，体验很差。流式方案让用户几乎实时看到文字逐个出现，感知到的等待时间大大缩短。

### 3.2 项目实现

**后端推送机制**

后端在接收到生成请求后，调用 DeepSeek 的流式 API。DeepSeek 每生成一个 token 就回调一次，后端立即将这个 token 包装成 SSE 格式推送给前端。

SSE 数据格式是 `data: {...}\n\n`，多个事件用两个换行符分隔。系统定义了三种事件类型：`start`（开始生成）、`content`（内容片段）、`done`（生成完成）、`error`（出错）。

**前端接收机制**

前端不能用普通的 axios 请求来接收 SSE，因为 axios 会等整个响应体接收完才回调。正确做法是用浏览器的 `fetch` API 配合 `ReadableStream`——边收边解析边渲染。

用户还可以随时点击"取消"按钮中断生成，底层通过 `AbortController` 实现，调用 `abort()` 方法后 fetch 立即中断，同时通知后端停止生成。

**关键 HTTP 头**

流式响应需要三个特殊的 HTTP 头：`Cache-Control: no-cache` 防止浏览器缓存流式内容，`Connection: keep-alive` 保持长连接，`X-Accel-Buffering: no` 告诉 Nginx 不要缓冲（否则 Nginx 会攒够全部内容再转发，流式效果就没了）。

## 四、YOLOv8 瑕疵检测系统

### 4.1 技术原理

**目标检测技术发展**

目标检测领域经历了几个阶段：R-CNN 系列（区域卷积神经网络）准确率高但速度慢，因为要逐区域扫描；YOLO 系列（You Only Look Once）将检测视为回归问题，整张图一次性预测所有目标框，速度极快。YOLOv8 是 ultralytics 整合的最新版本，一个框架同时支持分类、检测、分割。

**分类 vs 检测 vs 分割**

| 任务 | 输出 | 用途 |
|------|------|------|
| 分类 | 图片里有什么（一个标签） | 识别商品类别 |
| 检测 | 物体在哪（边界框） | 定位多个物体 |
| 分割 | 物体精确轮廓（像素级） | 精确画出瑕疵形状 |

**卷积神经网络基本原理**

CNN 通过卷积核（小窗口）在图片上滑动，提取局部特征（边缘、纹理、形状）。浅层提取简单特征（边缘），深层组合成复杂特征（物体部件）。YOLO 将整张图划分为网格，每个网格预测该区域的物体类别和边界框。

### 4.2 项目实现

**两个 YOLO 模型**

系统用了两个 YOLO 模型各司其职：分类模型识别商品属于哪个品类（iPhone 还是华为还是 MacBook），分割模型检测图片中的瑕疵并画出精确轮廓。

分类模型的输出是类别标签；分割模型的输出是像素级的瑕疵轮廓，可以精确计算瑕疵的面积，这对后续严重程度判断很关键。

**7 类 Kaputt 缺陷**

Kaputt 是德语"损坏"的意思，是一套工业级损伤分类标准。系统支持的 7 类缺陷是：穿透（屏幕被戳穿）、变形（机身摔弯）、功能故障（按键失效）、结构损坏（外壳开裂）、溢漏（屏幕或电池漏液）、表面瑕疵（划痕、掉漆、氧化）、部件缺失（少了按键或后盖）。

训练数据集中包含数千张标注了这些缺陷的商品图片。

**严重程度自动分级**

检测到瑕疵后，系统计算瑕疵的像素面积占整张图片的比例。占比超过 5% 为重度，1.5% 到 5% 为中度，0.3% 到 1.5% 为轻度，低于 0.3% 为轻微。

这个分级**不直接展示给用户**——因为用户看到"重度损伤"四个字可能直接放弃发布。系统只告诉用户检测到几个瑕疵，严重程度仅在内部传给 DeepSeek 做定价参考。

**标注图渲染**

根据严重等级使用不同的标注样式：重度用红色圆形框，中度用橙色矩形框，轻度用金色多边形，轻微用蓝色虚线框。然后用图像处理库在原图上画框并写中文标签。

中文标注需要字体文件，系统通过工具函数自动适配不同操作系统：Windows 用微软雅黑、macOS 用苹方、Linux 用文泉驿。

**单例模式加载模型**

模型权重文件加载需要 3-5 秒，占用约 200MB 内存。系统采用单例模式——整个进程只有一个检测器实例，首次加载后全局共享。如果每个请求都重新加载一次，服务器内存会迅速耗尽。

## 五、CLIP + Qdrant 向量搜索

### 5.1 技术原理

**CLIP 是什么？**

CLIP（Contrastive Language-Image Pre-training）是 OpenAI 提出的多模态模型，核心思想是将图片和文字映射到**同一个语义向量空间**。

训练时，CLIP 学习让匹配的图文对在向量空间中距离更近，不匹配的图文对距离更远。训练完成后，图片和文本都可以转换成 512 维向量，而且它们在同一空间里——相似的图片向量距离近，描述图片的文字和该图片的向量距离也近。

**ViT 是什么？**

ViT（Vision Transformer）把图片切成固定大小的小块（如 32×32），每个小块展平成一个向量，加上位置信息后，像处理文本单词序列一样用 Transformer 处理。这打破了 CNN 只能处理图像的局限。

**向量搜索原理**

传统搜索用关键词匹配，向量搜索用"语义距离"匹配。将查询内容转换为向量后，在向量空间中找最接近的 K 个向量。距离用余弦相似度衡量——越接近 1 表示越相似。

**Qdrant vs FAISS**

两者都是向量搜索引擎，但定位不同。FAISS 是 Facebook 出品的向量搜索库，优势是速度快，但它运行在内存中——进程重启索引就没了。Qdrant 是独立服务，数据持久化到磁盘，自带 HTTP API，支持过滤条件，更适合生产环境部署。

### 5.2 项目实现

**CLIP 编码器配置**

系统使用 `open_clip` 库加载 ViT-B-32 架构，预训练权重来自 LAION-2B 数据集（包含 20 亿图文对）。图片预处理流程是：缩放到 224×224、中心裁剪、标准化。文本预处理是分词、截断到 77 个 token。最终输出 512 维的浮点数向量。

**搜索流程**

用户上传图片后，CLIP 图片编码器提取 512 维向量，Qdrant 在索引中找出最相似的 Top-K 个向量 ID，同时支持按品类过滤（如只搜手机）。

得到 ID 列表后，用 MySQL 的 `IN` 查询一次性批量获取商品详情，而不是逐个查询——这叫避免 N+1 问题。

**索引自动构建**

用户每次保存发布时，后端异步执行：下载保存的图片、CLIP 提取向量、存入 Qdrant（附带标题、品类、价格作为元数据）。整个过程用户无感知。

**向量与关系数据的双写一致性**

保存时先写 MySQL（事务提交），再写 Qdrant。Qdrant 写入失败不影响 MySQL 数据，但会在日志中记录异常，后续可异步补偿。这是一个"最终一致性"的设计。

## 六、图像预处理管道

### 6.1 技术原理

**为什么需要预处理？**

用户拍照环境不可控——暗光、抖动、杂乱背景都会降低 AI 识别准确率。预处理的目的就是把不可控的原始图片统一处理成干净、清晰、标准尺寸的高质量输入。

**各步骤原理**

去噪：手机拍照的噪点是随机出现的彩色小点。去噪算法对每个像素取邻域像素的加权平均，能去除噪点同时保留真实边缘。

亮度增强（CLAHE）：将图片分成小网格，每个网格单独调整亮度分布，让暗的地方变亮、亮的地方不过曝。关键操作在亮度通道，不改变颜色。

对比度增强：线性拉伸像素值范围，让"灰蒙蒙"的图变得黑白分明。

锐化（USM）：先模糊得到"模糊版"，然后用原图减去模糊版得到"细节层"，再把细节层加回原图。效果是边缘更锐利。

等比缩放（Letterbox）：模型输入需要固定尺寸，如果直接拉伸会变形。Letterbox 等比缩放后用黑边填充短边，保证内容不变形。

主体检测：用边缘检测找图片中的轮廓，取最大轮廓作为主体，裁切掉无关背景。

### 6.2 项目实现

系统使用 OpenCV 实现上述 6 步管道。去噪用快速非局部均值去噪算法；亮度增强用 CLAHE 在 Lab 色彩空间的 L 通道操作；对比度增强用线性变换；锐化用 Unsharp Masking；缩放用 Letterbox 方式保证不拉伸。

主体检测基于 Canny 边缘检测找到最大轮廓作为主体区域。背景移除则是可选的，使用 rembg（基于 U²-Net 深度学习模型）做像素级背景分离，主要用于商城展示，让商品更突出。

实验表明，不做预处理时 Qwen-VL 的识别准确率下降约 30%。

## 七、数据飞轮

### 7.1 技术原理

**什么是数据飞轮？**

数据飞轮是 AI 系统持续改进的核心机制：系统运行时产生数据 → 数据用于训练模型 → 模型改进后系统效果更好 → 用户更多产生更多数据，形成正向循环。

**为什么需要数据飞轮？**

本地小模型（YOLO，几十 MB）速度快、免费，但准确率有限。云端大模型（Qwen-VL，千亿参数）准确率高，但每次调用要付费且有延迟。

能不能用大模型来"教"小模型，让小模型逐渐变准？这就是数据飞轮的核心思想——用云端大模型标注数据，积累下来训练本地小模型。

**模型蒸馏概念**

大模型"教"小模型的本质是知识蒸馏——大模型的输出包含了丰富的"软知识"（不仅告诉答案，还包含答案的置信度分布），小模型学习这些输出，相当于获取了大模型的判断模式。

### 7.2 项目实现

**双模型比对机制**

每次用户上传图片时，YOLO 和 Qwen-VL 同时推理。如果两者结果不一致，就记录到 `hard_cases` 表中，记录内容包括：图片 URL、YOLO 的错误分类、Qwen-VL 的正确分类、置信度等。

Qwen-VL 被视为"正确答案"的依据是：它是千亿参数多模态大模型，识别能力远超几十 MB 的 YOLO。虽非 100% 正确，但作为训练标注的参考值已足够。

**幂等设计**

同一张图、同一个错误、同一个正确标签可能被不同用户反复触发。表设计上用这三个字段的组合做唯一约束，保证同一错误组合只存一条记录，重复触发只累加计数。

**每周自动训练**

每周运行训练脚本，用 `hard_cases` 中积累的数据微调 YOLO 模型，生成新的权重文件替换旧版本，然后将这些 case 标记为已修复。

这样就形成了完整的闭环：用户触发不一致 → 记录到错题本 → 每周训练微调模型 → 新模型上线 → 准确性提升 → 需要云端模型纠正的案例减少 → 成本降低。

## 八、数据库设计

### 8.1 技术原理

**关系型数据库**

关系型数据库以表为单位存储数据，表之间通过外键关联。核心优势是 ACID 事务保证数据一致性，适合需要强一致性的业务数据。

**范式与冗余的权衡**

第三范式要求数据不冗余，但完全规范化的设计可能导致大量表关联查询，性能下降。实际工程中会适当冗余（如点赞数冗余计数器），换取查询性能，再通过事务保证冗余数据与源数据一致。

**JSON 列的应用场景**

当数据结构不确定或嵌套层级深时，JSON 列比拆分成多张表更灵活。缺点是 JSON 内部字段不能建立索引（除非用虚拟列），不适合做频繁条件查询的字段。

### 8.2 项目实现

**8 张表的关系**

系统共 8 张表，以 `published_items` 为核心。用户表关联商品发布表；商品表关联点赞记录、特征向量、错题本；用户表还关联 AI 审计日志。行情表和价格历史表相对独立。

**商品状态流转**

商品有三个状态：草稿、已发布、已下架。草稿可以转为已发布，已发布可以下架，已下架可以重新发布。

**点赞数双写策略**

点赞数据存两份：`item_likes` 表存真实的用户点赞记录（有唯一索引防止重复点赞），`published_items` 表的 `likes` 字段存冗余计数。

任何点赞操作都在一个事务里完成：先检查是否存在记录，存在则删除并减一，不存在则插入并加一。出错则回滚，保证两份数据始终一致。

**defect_data 用 JSON 列**

瑕疵数据包含类型、位置、面积、严重程度等多层嵌套信息。用 JSON 列存储，一条记录自包含所有瑕疵信息，未来新增损伤类型也不需要改表结构。

## 九、认证与安全体系

### 9.1 技术原理

**JWT 是什么？**

JWT（JSON Web Token）是一种无状态认证方案。服务器生成包含用户信息的 JSON 对象，用密钥签名后发给客户端。客户端后续请求携带该 Token，服务器验证签名即可确认用户身份，无需查询数据库。

JWT 的过期时间在 Token 本身中编码，服务器无状态，适合分布式部署。

**PBKDF2 是什么？**

PBKDF2（Password-Based Key Derivation Function 2）是一种密码哈希算法。它接受密码、随机盐值、迭代次数三个输入，输出固定长度的哈希值。

盐值防止彩虹表攻击（相同密码不同用户哈希不同），迭代次数增加计算成本。10 万次迭代让暴力破解变得极其昂贵——猜一个密码要花 0.1 秒，10 亿次尝试需要 3 年以上。

**滑动窗口限流原理**

固定窗口限流有边界漏洞：第 58 秒发 5 次请求、第 62 秒又发 5 次，虽然两秒内发了 10 次，但两个分钟窗口分别统计都不超限。

滑动窗口记录每个请求的时间戳，每次新请求来时，删除窗口外的旧记录，再统计窗口内的请求数。这是真正的"过去 60 秒内"限制，没有边界漏洞。

### 9.2 项目实现

**JWT 认证流程**

用户登录验证密码后，服务器生成 JWT（包含用户 ID 和过期时间），用 HS256 签名返回给前端。前端存入 localStorage，后续每个请求自动在 HTTP 头中携带。

后端通过依赖注入解析 Token，验证签名和过期时间，提取用户 ID 注入到路由参数。Token 无效或过期时自动返回 401，业务代码无需处理认证逻辑。

**前端拦截器处理 Token**

前端通过 Axios 请求拦截器自动从 localStorage 取 Token 加到请求头，响应拦截器检测 401 状态时自动清除 Token 并跳转到登录页。开发者不需要在每个接口调用里重复写这些逻辑。

**密码安全**

密码使用 PBKDF2-SHA256，加 16 字节随机盐，迭代 10 万次。存储格式为盐的十六进制加冒号加哈希值的十六进制。密码绝不存储明文，也无法逆向还原。

**身份防伪造**

所有需要认证的接口，用户 ID 都从 JWT 解析，完全忽略请求体里传来的 `user_id`。即使用户篡改请求体，也无法冒充其他用户。

**文件上传安全**

上传的图片先验证 MIME 类型（只允许图片格式），再用图像处理库实际解码验证内容是否为图片（防止伪装文件），最后对文件名做安全处理防止路径遍历攻击。

## 十、前端技术

### 10.1 技术原理

**Vue 3 响应式原理**

Vue 3 使用 Proxy 代理对象实现响应式。当数据变化时，Proxy 能捕获到变化并触发依赖更新，让视图自动重新渲染。

**Vite 构建原理**

Vite 利用浏览器原生 ESM（ES Module）支持，开发时不打包，直接按需加载源码文件。修改代码后只需替换被修改的模块，而非整个重新打包，热更新在毫秒级完成。

生产构建则用 Rollup 打包，兼顾开发体验和生产性能。

**SSE 接收原理**

SSE 基于 HTTP 长连接，浏览器通过 `EventSource` 或 `fetch` 建立连接，服务器持续发送 `data:` 前缀的数据块。连接可以保持数分钟甚至更长，直到服务器主动关闭或网络断开。

### 10.2 项目实现

**技术选型理由**

Vue 3 的模板语法和 Option API 对新手友好，团队规模小不需要 React 的复杂生态。Vite 的 ESM 原生开发带来毫秒级热更新，开发效率高。Tailwind 是原子化 CSS，提供完全的设计自由度，最终界面不像通用组件库那样千篇一律。

**三步骤发布流程**

首页设计为三步引导流程：第一步上传图片，自动调用视觉识别展示结果；第二步让用户确认或修正识别信息，并可选择查价和瑕疵检测；第三步点击生成，SSE 流式展示文案逐字出现，用户确认后一键保存发布。

每一步都有清晰的界面提示和进度反馈，降低用户的操作门槛。

**打字机效果渲染**

文案生成过程中，前端通过 SSE 接收到每个 token 后追加到页面，实现逐字出现的打字机效果。由于 token 推送频率很高，前端做了渲染节流——用 `requestAnimationFrame` 合并多个 token 后一次性更新 DOM，避免频繁操作导致性能问题。

**轻量状态管理**

项目只有登录状态需要全局共享（用户 ID、用户名、Token），直接用 Vue 3 的 `reactive()` 创建响应式对象即可满足需求。没有引入 Vuex 或 Pinia，避免不必要的复杂度。
