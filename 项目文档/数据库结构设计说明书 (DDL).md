# 数据库结构设计说明书 (DDL)

下面为你详细解剖这 4 张表的长相、具体作用以及工程设计意图：

### 1. `users` (用户信息表)

- **作用：** 系统的账号基石。所有在平台上生成二手商品、触发大模型调用的行为，都必须绑定到一个具体的“人”身上。
- **核心结构：**

| **字段名**   | **数据类型** | **作用说明**               |
| ------------ | ------------ | -------------------------- |
| `id`         | BIGINT       | 用户唯一标识（主键）。     |
| `username`   | VARCHAR      | 用户的昵称或账号名。       |
| `created_at` | DATETIME     | 这个账号是什么时候注册的。 |

- **工程视角：** 在目前的双人开发阶段，这主要是一个“占位表”。你们只需要在里面手动写死一个测试用户（比如 ID 为 1 的测试账号），后续所有的发布动作都默认算在这个测试账号头上，免去开发复杂的登录注册流程。

### 2. `market_prices` (二手行情基准表 —— 核心引擎)

- **作用：** 系统的“内部物价局”。大模型是不懂当前市场行情的（它会乱编价格），所以大模型只负责“认出”这是什么东西，然后后端拿着大模型认出的型号，来这张表里查真实的参考价。
- **核心结构：**

| **字段名**   | **数据类型** | **作用说明**                         |
| ------------ | ------------ | ------------------------------------ |
| `id`         | BIGINT       | 行情记录主键。                       |
| `category`   | VARCHAR      | 物品大类（例如：手机、外设、教材）。 |
| `brand`      | VARCHAR      | 品牌（例如：Apple、罗技）。          |
| `model`      | VARCHAR      | 具体型号（例如：iPhone 13、G610）。  |
| `low_price`  | DECIMAL      | 近期二手市场最低参考价。             |
| `high_price` | DECIMAL      | 近期二手市场最高参考价。             |

- **工程视角：** 这张表是**只读**的（对于 C 端用户而言）。开发前期，你们需要手动往这里录入几十条你们宿舍常见的物品价格数据。后端查询时，会频繁使用 `brand` 和 `model` 进行匹配。

### 3. `published_items` (商品发布记录表 —— 业务产物)

- **作用：** 记录最终的“工作成果”。当大模型把商品标题、带货文案写好，并结合数据库给出定价后，这些信息必须存下来，以便用户能在前端的“我的发布历史”页面中查阅。
- **核心结构：**

| **字段名**           | **数据类型** | **作用说明**                               |
| -------------------- | ------------ | ------------------------------------------ |
| `id`                 | BIGINT       | 发布记录主键。                             |
| `user_id`            | BIGINT       | 是谁发布的？（关联 `users` 表的 `id`）。   |
| `original_image_url` | VARCHAR      | 用户最初上传的那张旧物品照片的存储路径。   |
| `ai_generated_title` | VARCHAR      | AI 生成的吸睛标题（如：99新女生自用...）。 |
| `ai_generated_desc`  | TEXT         | AI 生成的详细带货文案。                    |
| `suggested_price`    | DECIMAL      | 系统结合行情表给出的最终建议定价。         |
| `status`             | TINYINT      | 当前状态（例如：1-待发布，2-已卖出）。     |

- **工程视角：** 这张表把 AI 的虚拟算力变成了具体的业务资产。前后端分离交互时，前端请求“历史记录”列表，后端就是从这张表里拿数据的。

### 4. `ai_audit_logs` (AI 调用风控与审计表 —— 面试亮点)

- **作用：** 系统的“黑匣子监控记录”。大模型的输出具有不可控性（可能报错、可能超时、可能输出违规词），这张表记录了系统与大模型交互的每一次详细过程，用于后续排查 Bug 和监控 API 成本。
- **核心结构：**

| **字段名**          | **数据类型** | **作用说明**                                 |
| ------------------- | ------------ | -------------------------------------------- |
| `id`                | BIGINT       | 日志主键。                                   |
| `user_id`           | BIGINT       | 触发这次大模型调用的用户 ID。                |
| `action_type`       | VARCHAR      | 是什么动作？（例如：视觉提取、生成文案）。   |
| `raw_ai_response`   | **JSON**     | 大模型返回的**最原始、最完整**的内容。       |
| `execution_time_ms` | INT          | 这次大模型思考了多久（毫秒）。用于性能分析。 |
| `status`            | VARCHAR      | 成功（SUCCESS）还是失败报错（FAILED）。      |

- **工程视角：** 这是标准的企业级应用开发规范。把大模型返回的内容存在 JSON 类型的字段里，即使以后大模型的输出格式变了，也不需要去修改表结构（加减列），极具扩展性。

这四张表紧密咬合：**用户（表1）上传图片 $\rightarrow$ AI 提取特征并去查价（表2） $\rightarrow$ 生成的内容存入记录（表3） $\rightarrow$ 整个过程的耗时与细节被悄悄记入日志（表4）。**
