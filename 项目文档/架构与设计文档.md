### 1.2 技术架构分层详解

#### 1.2.1 前端层

| 技术 | 版本 | 用途 |
|------|------|------|
| Vue 3 | ^3.5.13 | SPA 框架，使用 Composition API (`<script setup>`) 编写全部组件 |
| Vite | ^6.0.7 | 开发服务器 + 构建工具，HMR 热更新 |
| TailwindCSS | ^3.4.17 | 原子化 CSS 框架，响应式布局 |
| Chart.js + vue-chartjs | ^4.4.1 / ^5.3.0 | 价格历史折线图绘制 |
| Axios | ^1.7.9 | HTTP 客户端，baseURL 为 `/api/v1`，含请求/响应拦截器 |
| Vue Router | ^4.5.0 | 前端路由，history 模式，含路由守卫 |
| Pinia (reactive store) | (内置于 Vue 3) | 全局用户状态管理（user.js） |

**构建产出**：由 `vite build` 构建到 `backend/static/dist/`，产物包括：
- `index.html`：入口 HTML
- `assets/vue-vendor-*.js`：Vue + Vue Router + Axios 打包为一个 vendor chunk
- `assets/index-*.js`：各页面按需加载的代码块
- `assets/index-*.css`：TailwindCSS 产物

**开发代理**：`vite.config.js` 中将 `/api` 和 `/static` 代理到 `http://localhost:8000`，避免跨域问题。

#### 1.2.2 网关层

- **开发环境**：Vite Dev Server（`127.0.0.1:5173`），内置 proxy 转发到 FastAPI `:8000`
- **生产环境**：Nginx 反向代理，承担：
  - SSL/TLS 终端
  - 静态资源缓存
  - 请求转发到 Uvicorn（`0.0.0.0:8000`）
  - 添加 HSTS 安全头

#### 1.2.4 AI 模型层

| 模型 | 部署方式 | 设计模式 | 配置 |
|------|---------|---------|------|
| **YOLOv8**（商品识别） | 本地 CPU/GPU | **线程安全单例**（`threading.Lock` + 双重检查） | `best.pt`, `yolov8n.pt` |
| **YOLOv8**（瑕疵检测） | 本地 CPU/GPU | **线程安全单例** | `defect_best.pt`, Kaputt 7类缺陷 |
| **CLIP ViT-B-32** | 本地 CPU/GPU | **线程安全单例**（`threading.Lock` + 双重检查） | `open_clip`, `laion2b_s34b_b79k`, 512维 |
| **Qwen-VL-Max** | 阿里云百炼 API | **策略模式** LLM 客户端 | `dashscope`, `qwen-vl-max` |
| **DeepSeek** | DeepSeek API | **策略模式** LLM 客户端（单例） | `deepseek-chat`, 60s timeout |

**双通道容错机制**：
- YOLO（本地）与 Qwen-VL（云端）同时识别，YOLO 结果作为上下文喂给 Qwen-VL
- Qwen-VL 做语义级裁决，判断 YOLO 是否正确（`yolo_correct` 字段）
- 任一模型失败不影响另一模型结果，有完整的 fallback 路径

### 1.3 项目模块划分

系统按功能域划分为 6 大模块：

#### 模块 1：用户认证模块（auth）

| 角色 | 文件 |
|------|------|
| 路由 | `app/routers/auth.py` |
| JWT 处理 | `app/auth/jwt_handler.py` |
| 密码哈希 | PBKDF2-SHA256（600,000 次迭代，OWASP 2026 标准） |
| 依赖注入 | `app/dependencies.py`（`get_current_user`, `get_current_user_optional`, `get_current_admin`） |
| 前端 | `LoginPage.vue`, `RegisterPage.vue`, `store/user.js` |

**功能**：
- 用户注册（用户名/密码，自动哈希）
- 用户登录（返回 JWT，7天有效期）
- JWT 解析与验证（HS256 算法）
- 强制/可选/管理员三种认证依赖
- 前端 Axios 拦截器自动附加 `Authorization: Bearer <token>`
- 401 响应自动清除 token 并重定向登录页

#### 模块 2：商品识别模块（recognition + extract）

| 角色 | 文件 |
|------|------|
| 核心路由 | `app/routers/recognition.py`（`POST /recognize`，全链路入口，~780行） |
| 提取路由 | `app/routers/extract.py`（`POST /extract`，Qwen-VL 单独调用） |
| 视觉服务 | `app/services/vision_service.py`（JSON 解析，两层防护） |
| YOLO 检测器 | `app/ml/yolo_detector.py` |
| Qwen-VL 客户端 | `app/llm/qwen_vl_client.py` |
| 预设类别 | `app/constants/categories.py`（80类预设商品 + 品牌型号映射） |
| 前端 | `HomePage.vue`, `ImageUploader.vue` |

**依赖的模型**：YOLOv8（本地）、Qwen-VL-Max（云端阿里云百炼）

**核心流程**：
1. 图片上传 → 文件校验（MIME + PIL）
2. 图片预处理（6步管道：缩放→多物体判断→去背景→去噪→缩放至448×448）
3. 并行：YOLO 检测 + 瑕疵检测
4. YOLO 结果作为上下文给 Qwen-VL → Qwen-VL 语义识别
5. 双模型比对：Qwen-VL 判断 YOLO 是否正确
6. 非80类商品自动存入错题集（hard_cases + model_disagreements）

#### 模块 3：瑕疵检测模块（process + damage_detector）

| 角色 | 文件 |
|------|------|
| 路由 | `app/routers/process.py`（`POST /process/image`） |
| 瑕疵检测器 | `app/ml/defect_detector_yolo.py` |
| 成色判定 | `app/services/damage_detector.py`（5级成色程度计算） |
| 图像处理 | `app/services/image_processor.py` |

**依赖的模型**：YOLOv8 缺陷检测模型（`defect_best.pt`，Kaputt 7 类缺陷）

**7 类缺陷**：穿透、变形、功能故障、结构损坏、溢漏、表面瑕疵、部件缺失

**5 级成色**：完整（无瑕疵）→ 轻微瑕疵 → 中度瑕疵 → 重度瑕疵 → 完全损坏

#### 模块 4：文案生成模块（generate + text_service）

| 角色 | 文件 |
|------|------|
| 路由 | `app/routers/generate.py`（`POST /generate`，SSE 流式） |
| 文案服务 | `app/services/text_service.py`（Prompt 构建 + 流式调用） |
| DeepSeek 客户端 | `app/llm/deepseek_client.py` |
| 前端 | `TypewriterText.vue`（打字机效果渲染） |

**依赖的模型**：DeepSeek（云端 API）

**核心流程**：
1. 前端发送商品信息 + 市场行情
2. 后端构建 System Prompt（带货专家人设）+ User Prompt
3. DeepSeek 流式 API 逐 token 返回
4. 后端以 SSE（`text/event-stream`）推送给前端
5. 前端 `ReadableStream` 解析 → `requestAnimationFrame` 节流 → 打字机效果

#### 模块 5：商城与搜索模块（market + search）

| 角色 | 文件 |
|------|------|
| 商城路由 | `app/routers/market.py`（`GET /market`，分页+排序+筛选） |
| 搜索路由 | `app/routers/search.py`（`POST /search/image`, `/text`, `/by-filter`, `/semantic`） |
| CLIP 提取器 | `app/ml/clip_extractor.py` |
| Qdrant 客户端 | `app/ml/qdrant_client.py` |
| 前端 | `MarketPage.vue`, `SearchPage.vue`, `PriceHistoryPage.vue` |

**依赖的模型**：CLIP ViT-B-32（本地，512维向量）、DeepSeek（语义搜索）

**搜索方式**：
- **以图搜图**：上传图片 → CLIP 提取 512 维向量 → Qdrant HNSW 检索 → MySQL 批量查详情
- **以文搜图**：输入文字 → CLIP 文本编码 → Qdrant 检索
- **条件筛选**：品类/品牌/型号级联选择 → MySQL 直接查询
- **语义搜索**：自然语言描述 → DeepSeek 语义匹配 → 返回结果

#### 模块 6：系统管理模块（admin + notifications）

#### 关键步骤详解

**步骤 ① — 文件校验（`file_validator.py`）**

| 校验项 | 方法 | 失败处理 |
|--------|------|---------|
| 文件名安全性 | 检查 `..`, `/`, `\`, 控制字符 | 400 Bad Request |
| MIME 类型 | 白名单：JPEG/PNG/GIF/WebP/BMP/TIFF | 警告不阻断（浏览器 MIME 不可靠） |
| 扩展名 | 白名单：`.jpg/.jpeg/.png/.gif/.webp/.bmp/.tiff/.tif` | 400 Bad Request |
| 图片有效性 | PIL `Image.open()` + `verify()` | 400 Bad Request |
| 文件大小 | ≤ `MAX_UPLOAD_SIZE` (默认 10MB)，流式读取防 OOM | 400 Bad Request |
| 图片尺寸 | ≤ 4096×4096 | 400 Bad Request |

**步骤 ③ — 图片预处理（6 步管道）**

1. `resize(target_size=448)` → 统一为 448×448
2. Canny 边缘检测 → 找物体轮廓
3. 多物体判断 → 选择面积最大的物体（面积比 > 60% 才通过）
4. `rembg.remove()` → AI 去背景
5. `cv2.GaussianBlur(5,5)` → 高斯去噪
6. 最终缩放至 448×448 → 喂给 YOLO

**步骤 ⑥ — 双模型比对**

**步骤 ⑧ — DeepSeek 定价规则**

| 成色程度 | 定价策略 | 示例 |
|---------|---------|------|
| 完整（无瑕疵） | 市场均价上浮 5-10% | 均价 3000 → 建议 3150-3300 |
| 轻微瑕疵 | 按市场均价 | 均价 3000 → 建议 3000 |

#### 异常处理分支

```
识别流程异常分支:

Level 1: 文件校验失败
  → HTTP 400 "文件验证失败: {具体原因}"
  → 前端显示错误提示

Level 2: 预处理失败
  → 警告日志但强制缩放到 448，继续后续流程
  → "预处理警告: {msg}"

Level 3: YOLO 识别异常
  → catch Exception → { success: false, category: 'unknown' }
  → 不影响 Qwen-VL 继续运行

Level 4: Qwen-VL 调用失败
  → HTTP 422 "视觉大模型调用失败，请稍后重试"
  → 阻断流程（核心识别不可缺）

Level 5: Qwen-VL 返回无效 category
  → blocked = {'未知', 'unknown', '其他', 'other', ...}
  → HTTP 422 "视觉大模型未能识别商品品类"

Level 6: DeepSeek 调用失败
  → 兜底价格 = market_avg_price || _estimate_price(category)
  → 不影响返回结果结构

Level 7: 审计日志/通知失败
  → catch Exception，记录日志
  → 不影响主流程响应
```

#### 预估耗时

| 步骤 | 预估耗时 | 说明 |
|------|---------|------|
| 文件校验 | ~50ms | 流式读取 + PIL 验证 |
| 图片保存 | ~20ms | 异步文件写入 |
| 预处理（6步） | ~500ms | rembg 去背景最耗时 |
| YOLO 检测 | ~200ms | 单张图片推理 |
| 瑕疵检测 | ~200ms | 与 YOLO 并行执行 |
| Qwen-VL 识别 | 2~5s | API 网络调用，取决于图片大小 |
| 市场行情查询 | ~50ms | MySQL 查询 |
| DeepSeek 文案+定价 | 3~8s | API 网络调用 |
| **总计** | **约 6~14s** | 瓶颈在云端 API 调用 |

**SSE 事件类型**：

| type | 含义 | 数据结构 |
|------|------|---------|
| `start` | 生成开始 | `{"type":"start","status":"started","message":"开始生成文案..."}` |
| `content` | 文本片段（逐 token） | `{"type":"content","content":"这是..."}` |
| `done` | 生成完成 | `{"type":"done","status":"success","message":"文案生成完成"}` |
| `error` | 生成失败 | `{"type":"error","status":"error","code":"GENERATION_ERROR","message":"..."}` |

**关键技术点**：
- 响应头：`Cache-Control: no-cache`, `Connection: keep-alive`, `X-Accel-Buffering: no`（禁用 Nginx 缓冲）
- 前端使用原生 `fetch()` 替代 EventSource（EventSource 不支持 POST + JSON body）
- `AbortController` 支持用户取消生成
- `requestAnimationFrame` 节流防止频繁 DOM 更新导致卡顿

### 4.3 索引策略

| 表 | 索引 | 类型 | 设计理由 |
|----|------|------|---------|
| `users` | `username` | UNIQUE | 登录时按用户名查询，必须唯一且快速 |
| `users` | `idx_status` | INDEX | 管理员按状态筛选用户（active/disabled） |
| `users` | `idx_role` | INDEX | 管理员按角色筛选（user/admin） |
| `published_items` | `idx_user_id` | INDEX | 查询"我的发布历史"的主过滤条件 |
| `published_items` | `idx_status` | INDEX | 商城只展示 `published` 状态商品 |
| `published_items` | `idx_category` | INDEX | 按品类筛选是最常见的搜索行为 |
| `published_items` | `idx_created_at` | INDEX | 商城和历史列表默认按时间倒序 |
| `published_items` | `idx_status_category` | 联合索引 | 覆盖"已发布+品类筛选"的查询场景 |
| `published_items` | `idx_brand_model` | 联合索引 | 价格查询的三层匹配策略依赖 brand+model |
| `published_items` | `idx_status_category_brand` | 联合索引 | 覆盖最常见的多条件筛选 SQL |
| `market_prices` | `uk_brand_model` | UNIQUE | 行情数据的核心查询键，同时防止重复 |
| `market_prices` | `idx_category` | INDEX | 按品类聚合市场行情 |

### 5.3 路由守卫设计

**路由配置**（Vue Router history 模式）：

| 路由路径 | 页面组件 | 认证要求 | 说明 |
|---------|---------|---------|------|
| `/` | HomePage | 无 | 首页（图片上传+识别） |
| `/home` | HomePage | 无 | 首页别名 |
| `/market` | MarketPage | 无 | 商城浏览 |
| `/search` | SearchPage | 无 | 搜索页面 |
| `/history` | HistoryPage | **需要登录** | 仅展示已登录用户的历史 |
| `/price-history` | PriceHistoryPage | 无 | 价格走势查询 |
| `/notifications` | NotificationPage | **需要登录** | 仅展示已登录用户的通知 |
| `/admin` | AdminPanel | **管理员** | 系统管理 |
| `/login` | LoginPage | 无 | 登录页面 |
| `/register` | RegisterPage | 无 | 注册页面 |

**路由守卫实现逻辑**：

1. **前端守卫**：`router.beforeEach()` 检查 `localStorage.getItem('token')` 是否存在，若无 token 且目标页面需要认证，则重定向到 `/login`。
2. **后端守卫**：所有需要认证的 API 通过 `Depends(get_current_user)` 进行 JWT 校验，user_id 从 JWT 的 `sub` 字段提取，不可伪造。
3. **管理员路由**：额外使用 `Depends(get_current_admin)` 检查 `role == 'admin'`，否则返回 403。
4. **SPA 回退**：FastAPI 中的 `spa_fallback` 路由将所有非 API/非静态文件请求返回 `index.html`，支持前端 Vue Router history 模式下的页面刷新。

### 6.2 防护机制

#### 6.2.1 SQL 注入防护

| 措施 | 实现 | 覆盖范围 |
|------|------|---------|
| 全参数化查询 | `cursor.execute("... WHERE id=%s", (id,))` | **100% 的 SQL 操作** |
| ORDER BY 白名单 | `_MARKET_SORT_COLUMNS = {"created_at": "p.created_at", "price": "p.suggested_price"}` | 商城排序参数 |
| 排序字段白名单 | `_SORT_COLUMNS = {"created_at": ..., "retry_count": ..., "confidence": ..., "id": ...}` | 硬案例查询排序 |
| LIKE 参数化 | `params.append(f"%{keyword}%")` → `cursor.execute("... LIKE %s", (kw,))` | 所有模糊搜索 |
| 动态 SQL 防护 | 从不在 SQL 中拼接用户输入，始终使用 `%s` 占位符 | 所有动态构建的查询 |

#### 6.2.2 文件上传安全

| 校验层 | 方法 | 阻断策略 |
|--------|------|---------|
| 文件名安全检查 | 检测 `..`, `/`, `\`, `\x00`, 控制字符 | 直接拒绝 (400) |
| MIME 类型检查 | 白名单：JPEG/PNG/GIF/WebP/BMP/TIFF | 警告不阻断（浏览器 MIME 不可靠） |
| 扩展名检查 | 白名单：`.jpg/.jpeg/.png/.gif/.webp/.bmp/.tiff/.tif` | 直接拒绝 (400) |
| PIL 内容验证 | `Image.open()` + `verify()` + 格式/尺寸检查 | 直接拒绝 (400) |
| 文件大小限制 | 流式读取，实时检查累计大小 ≤ 10MB | 直接拒绝 (400) |
| 图片尺寸限制 | ≤ 4096×4096 px | 直接拒绝 (400) |
| 安全文件名生成 | 移除危险字符，限制 ASCII，截断过长名称 | 自动处理 |
