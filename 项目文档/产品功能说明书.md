## 一、产品概述

### 1.1 产品定位

智能二手商品发布助手是一款面向 C 端用户的 AI 驱动二手商品发布平台。用户只需上传一张商品照片，AI 即可自动完成品类识别、市场查价、瑕疵检测、智能定价、文案生成全流程，最终一键发布到商城。

**一句话介绍：** 拍一张照片，AI 帮你搞定二手商品发布的一切。

本产品的核心价值在于：**将传统二手商品发布中 10-20 分钟的人工操作压缩到 1 分钟以内**，同时利用 AI 的多维度分析能力，帮助用户做出更合理的定价决策，写出更专业的商品文案。

产品涵盖三大核心能力矩阵：

| 能力域 | 技术手段 | 用户可感知的价值 |
|--------|---------|----------------|
| **视觉理解** | Qwen-VL-Max + YOLOv8 双模型协同 | 上传图片即自动识别商品品类、品牌、型号、成色 |
| **市场智能** | MySQL 行情查价 + DeepSeek 定价推理 | 结合市场行情和瑕疵程度给出合理建议价 |
| **内容生成** | DeepSeek SSE 流式文案生成 | 200 字专业带货文案，逐字呈现的打字机效果 |

### 1.2 目标用户

本产品的目标用户分为三类：

**第一类：个人闲置物品卖家（学生群体为主）**

这是产品的核心用户群。大学生在毕业季、换季时有大量闲置电子产品需要处理（手机、笔记本、平板、耳机、手表、键盘等）。他们普遍面临"不知道卖多少钱"和"不会写商品描述"两大痛点。传统的解决方式是打开闲鱼或转转手动比价，然后绞尽脑汁写描述——耗时长、效果差。

本产品对这类用户的价值：上传照片后 AI 自动搞定一切，1 分钟内完成发布，极大降低二手交易门槛。

**第二类：二手商品交易新手**

这类用户之前从未发布过二手商品，对市场行情、商品成色描述、定价策略、文案撰写完全没有概念。他们需要的是一个"导购式"的智能工具，能够一步步引导他们完成发布。

本产品对这类用户的价值：AI 在每个环节给出建议（品类是什么、市场价多少、瑕疵有多严重、建议卖多少钱、应该怎么描述），用户只需要确认和微调即可。

**第三类：需要批量处理商品的二手商家**

二手数码商家每天需要处理大量商品的入库、检测、定价、上架工作。虽然他们经验丰富，但人工操作的效率瓶颈依然存在。批量上传 + AI 自动处理可以大幅提升运转效率。

本产品对这类用户的价值：批量上传图片 → 批量 AI 识别 → 一键生成多件商品，大幅节省人力成本。

### 1.3 产品愿景

本产品的长期愿景是：**让每个人的闲置物品都能被精准展示、合理定价、快速流转**。通过 AI 技术消除信息不对称——卖方不需要是鉴定专家也能描述清楚商品状况，买方通过以图搜图和 AI 分析能了解真实市场行情。

## 二、应用场景与痛点分析

### 2.1 传统二手交易流程痛点

在开发本产品之前，我们对 50 名大学生进行了二手交易习惯调研，总结出以下五大核心痛点：

| 痛点 | 传统做法 | 用户困扰 | 本产品解决方案 |
|------|---------|---------|---------------|
| **不知道卖多少钱** | 手动去闲鱼、转转翻半天比价，价格要么太高卖不掉，要么太低亏了 | 比价效率极低；相同商品不同卖家定价差异巨大，缺乏参考坐标；个人对行情判断不准 | AI 自动查市场行情（品牌+型号精准匹配），结合瑕疵程度智能定价，给出有数据支撑的合理建议价，并附带扣款明细说明 |
| **不会写商品描述** | 绞尽脑汁凑字数，标题不吸引人，关键信息遗漏（如版本、颜色、配件情况），描述缺乏说服力 | 描述干瘪无吸引力；关键参数容易遗漏导致买家反复咨询；不知道如何突出卖点；花大量时间仍写不好 | DeepSeek 大语言模型自动生成专业带货文案，自动包含关键卖点与产品参数，200 字精品文案，配合 SSE 流式打字机效果呈现 |
| **看不出商品瑕疵** | 肉眼检查容易遗漏，描述与实际不符导致交易纠纷；到货后买家发现未标注的划痕，要求退货或部分退款 | 对手机屏幕细微划痕、机身磕碰等缺陷判断不准确；描述与实际不符引发纠纷；作为新手卖家缺乏鉴定经验 | YOLOv8-seg 分割模型自动检测 7 类 Kaputt 缺陷（穿透、变形、功能故障、结构损坏、溢漏、表面瑕疵、部件缺失），4 级严重程度自动分级，在原图上画框标注展示 |
| **商品太多发布慢** | 逐一拍照、想文案、填信息（品类、品牌、型号、价格、描述），一件商品耗时 10-20 分钟。如果有多件商品需要出售，整个过程非常枯燥 | 多件商品逐一发布极其费时；重复性劳动令人烦躁；繁琐的填写项让人放弃 | 上传 → AI 全自动 → 确认 → 发布，全流程 1 分钟完成。支持草稿保存与批量管理 |
| **搜不到相似商品** | 纯文字搜标题，匹配不准确，想找个参考价都难；输入"iPhone"可能搜出一堆手机壳和贴膜 | 关键词匹配太宽泛；搜不到视觉上相似的参考商品；无法按外观找同类 | 以图搜图 + 以文搜图：上传图片直接找到视觉上最像的商品，附带价格参考；输入文字描述也能按语义检索 |

### 2.2 典型使用场景

**场景一：大学生毕业季甩卖电子产品**

> 张同学即将毕业离校，手头有一台使用了两年的 iPhone 14、一副 AirPods Pro、一个 iPad Air。他需要尽快处理掉这些电子产品，但对二手行情完全不了解。
>
> **使用本产品：**
> 1. 上传 iPhone 14 照片 → AI 自动识别为"Apple iPhone 14 128GB 午夜色"，成色 8 成新
> 2. 瑕疵检测：屏幕有 2 处中度划痕，边框 1 处轻度磕碰
> 3. 市场查价：同型号均价 ¥3,200，最低 ¥2,800，最高 ¥3,800
> 4. 智能定价：考虑瑕疵扣除 ¥400，建议售价 ¥2,800
> 5. AI 文案：自动生成一段包含"2023年购入、午夜色、128GB、屏幕细微划痕、功能完好"的文案
> 6. 一键发布到商城，3 分钟内完成全部操作
>
> 三件商品总共花费不到 10 分钟，而传统方式至少需要 1 小时。

**场景二：数码爱好者升级设备，出手旧款**

> 李同学是数码发烧友，经常升级设备，需要出手旧款设备来减轻经济压力。他对电子产品比较了解，但写文案仍然头疼。
>
> **使用本产品：**
> 1. 上传旧款 MacBook Pro 照片 → AI 识别为"Apple MacBook Pro 14 M2 Pro"
> 2. 检测到 A 面有 1 处轻微划痕，键盘轻微油光
> 3. 市场查价：均价 ¥12,000，结合瑕疵建议 ¥11,000
> 4. 一键生成专业文案，突出 M2 Pro 性能和 32GB 内存
> 5. 以图搜图发现同类 MacBook 售价区间 ¥10,000-13,000，确认定价合理
>
> AI 文案比李同学自己写得更好，突出了专业参数和卖点。

**场景三：二手商家批量上架商品**

> 王老板经营一家二手数码店，每天需要入库处理数十台手机和笔记本。他精通行情但人工操作瓶颈严重。
>
> **使用本产品：**
> 1. 批量上传多台手机照片
> 2. AI 自动识别每台设备的品牌型号和成色
> 3. 瑕疵检测筛选出需要降价的"问题机"
> 4. 批量生成文案并一键发布
> 5. 以图搜图功能帮助顾客在店内快速找到同类商品
>
> 相比人工逐台上架，效率提升 5 倍以上。

## 三、核心功能清单

### 3.1 商品发布（核心链路）

商品发布是系统的核心业务流程，涵盖从图片上传到最终发布的完整闭环。

| 功能 | 说明 | 技术实现 | 用户感知 |
|------|------|---------|---------|
| 图片上传 | 支持 JPG/PNG/WEBP 格式，拖拽或点击上传，单张最大 10MB | FastAPI + 文件安全校验（MIME 验证 + PIL 解码验证 + 路径遍历防护） | 拖入图片即开始处理，无需额外操作 |
| 图像预处理 | 6 步自动预处理管道提升图片质量 | 448×448 尺寸统一 → rembg 去背景 → OpenCV CLAHE 亮度增强 → 对比度拉伸 → USM 锐化 → 主体检测 | 用户无感知，后台自动优化拍摄质量差的照片 |
| AI 视觉识别 | 自动识别品类、品牌、型号、成色 | Qwen-VL-Max 多模态大模型进行结构化识别，输出 JSON（品类、品牌、型号、成色描述） | 识别结果直接展示，用户可修正 |
| YOLO 物品检测 | 检测图片中的物品类别和位置 | YOLOv8 预训练模型（80 类 COCO 类别），多色画框标注 + Base64 输出 | 了解图片中识别到了哪些物品 |
| 双模型比对 | YOLO 本地模型 vs Qwen-VL 云端模型并行识别 | 两个模型同时推理，结果比对，不一致自动收集到错题本 | 提升识别准确率和可靠性 |
| 瑕疵检测 | 7 类 Kaputt 缺陷自动检测 + 4 级严重程度分级 | YOLOv8-seg 分割模型进行像素级缺陷检测 | 检测出划痕、磕碰等缺陷并在图上标注 |
| 市场查价 | 品牌+型号查询二手均价、最低价、最高价 | MySQL 三层匹配策略（精确匹配 → 模糊匹配 → 返回空） | 了解同类商品市场行情参考 |
| 智能定价 | 结合瑕疵程度 + 市场行情给出建议售价 + 扣款明细 | DeepSeek 大语言模型推理 | 获得有理有据的建议售价 |
| 文案生成 | 逐字流式输出带货文案，打字机效果 | DeepSeek SSE 流式生成 | 实时看到文案逐字出现，体验流畅 |
| 一键发布 | 保存到商城 + 自动加入以图搜图索引 | MySQL 事务保存 + Qdrant 向量入库 | 商品即刻进入商城展示 |
| 草稿管理 | 保存草稿、发布草稿、下架商品、重新发布 | status 字段状态流转（draft → published → delisted） | 灵活管理发布节奏 |

### 3.2 商城系统

用户发布的所有商品在商城中集中展示，支持浏览、搜索、筛选。

| 功能 | 说明 | 技术实现 |
|------|------|---------|
| 商品列表 | 已发布商品按时间倒序展示，最多 100 条 | published_items 表 WHERE status='published'，ORDER BY created_at DESC |
| 关键词搜索 | 匹配商品标题和描述内容 | MySQL LIKE 模糊查询 |
| 品类筛选 | 按手机、笔记本、平板、耳机、手表、外设等品类过滤 | WHERE category = ? |
| 浏览计数 | 每次点击详情页浏览次数 +1 | UPDATE views = views + 1 |
| 商品详情 | 展示完整商品信息（标题、描述、价格、瑕疵标注图、发布者） | 单表查询 + JOIN users |
| 响应式设计 | 适配桌面端和移动端 | TailwindCSS 响应式栅格 |

### 3.3 以图搜图

突破传统文字搜索的局限，通过图像语义理解实现"以图搜图"和"以文搜图"。

| 功能 | 说明 | 技术实现 |
|------|------|---------|
| 以图搜图 | 上传图片 → CLIP 特征提取 → Qdrant 向量检索 → 返回相似商品及相似度分数 | CLIP ViT-B-32 图片编码器（512 维向量）+ Qdrant HNSW 索引 |
| 以文搜图 | 输入文字描述 → CLIP 文本编码 → 向量检索 → 返回匹配商品 | CLIP 文本编码器 + Qdrant |
| 自动索引 | 商品发布时自动提取特征并存入 Qdrant | 异步后台任务 |
| 自动清理 | 商品下架时自动从 Qdrant 删除对应向量 | 保证搜索结果的有效性 |
| 品类过滤 | 搜索结果可按品类筛选（如只搜手机类） | Qdrant filter 条件 |
| Top-K 控制 | 限制搜索返回结果数量，防止资源滥用 | 参数校验上限 100 |

### 3.4 价格历史

为每款商品维护价格历史记录，提供趋势分析和 AI 预估。

| 功能 | 说明 | 技术实现 |
|------|------|---------|
| 价格走势图 | Chart.js 绘制 180 天价格曲线 | price_history 表查询 + Chart.js 折线图 |
| 价格快照 | 定期记录品牌+型号+价格+时间 | INSERT INTO price_history |
| 价格类型 | 区分均价、最低价、最高价 | price_type 枚举（avg/low/high） |
| AI 预估 | 基于历史数据估算当前合理价格 | history_estimator 服务 |
| 涨跌分析 | 最高价/最低价/均价统计，展示价格波动范围 | SQL 聚合查询 |

### 3.6 管理后台

为管理员提供的系统管理功能。

| 功能 | 说明 | 技术实现 |
|------|------|---------|

## 四、功能详细说明

### 4.1 用户注册与登录

**注册流程：**

1. 用户输入用户名和密码（密码最少 6 位）
2. 系统检查用户名是否已存在（用户名唯一约束）
3. 使用 PBKDF2-SHA256 算法对密码进行哈希处理：
   - 生成 16 字节的随机盐值
   - 使用 100,000 次迭代的 PBKDF2 算法
   - 存储格式：`hex(salt_16bytes):hex(hash_32bytes)`
4. 插入用户记录，默认状态为 `active`，默认角色为 `user`
5. 返回注册成功，引导用户登录

**登录流程：**

1. 用户输入用户名和密码
2. 系统查询用户记录，验证密码哈希
3. 验证通过后生成 JWT Token：
   - 签名算法：HS256
   - Payload：user_id + exp（过期时间）
   - 使用 SECRET_KEY 进行签名
4. 更新 last_login 时间
5. 返回 JWT Token 给前端
6. 前端存入 localStorage，后续请求自动携带

**JWT 认证中间件：**

- 每个需要认证的 API 端点通过 FastAPI 依赖注入自动验证 Token
- Token 无效或过期时自动返回 401 Unauthorized
- 从 Token 中提取 user_id，忽略请求体中的 user_id，防止身份伪造

**账户安全策略：**

- 注册限流：5 次/分钟（滑动窗口）
- 登录限流：10 次/分钟（滑动窗口）
- 密码不可逆哈希，忘记密码只能重置
- 管理员可通过管理后台封禁/解封用户

### 4.2 图片上传与预处理

**上传方式：**

- 支持点击选择文件和拖拽上传两种方式
- 支持格式：JPG、PNG、WEBP
- 单个文件最大：10MB
- 上传组件提供实时预览和进度反馈

**安全检查（服务器端）：**

1. **MIME 类型验证**：检查 Content-Type 头，只允许 image/jpeg、image/png、image/webp
2. **PIL 解码验证**：用 Pillow 库实际解码图片内容，确认真的是图片（防止将恶意脚本伪装成图片）
3. **文件名安全处理**：过滤路径分隔符，防止路径遍历攻击（如 `../../../etc/passwd`）
4. **文件大小限制**：检查上传文件大小不超过 10MB

**6 步预处理管道：**

用户拍照环境不可控——暗光、抖动、杂乱背景都会降低 AI 识别准确率。系统在上传后自动执行以下预处理：

| 步骤 | 操作 | 目的 | 技术实现 |
|------|------|------|---------|
| 1 | 尺寸统一 | 确保模型输入格式一致 | 等比缩放到 448×448（Letterbox），不变形 |
| 2 | 去背景 | 移除杂乱背景，突出商品主体 | rembg（基于 U²-Net 深度学习模型）像素级背景分离 |
| 3 | 去噪 | 去除手机拍照的噪点 | OpenCV 快速非局部均值去噪（fastNlMeansDenoising） |
| 4 | 亮度增强 | 让暗处细节可见 | CLAHE（限制对比度自适应直方图均衡化），在 Lab 色彩空间 L 通道操作 |
| 5 | 对比度增强 | 消除灰蒙感 | 线性拉伸像素值范围 |
| 6 | 锐化 | 增强边缘锐利度 | USM（Unsharp Masking）：原图 - 模糊图 = 细节层，叠加回原图 |

**实验数据：**

实验表明，经过预处理后的图片，Qwen-VL 的识别准确率提升约 30%，YOLO 的检测置信度提升约 15%。

### 4.3 AI 视觉识别

**Qwen-VL-Max 识别：**

Qwen-VL-Max 是阿里云百炼平台的千亿参数多模态视觉语言大模型，能够同时理解图像内容和文字指令。系统使用它对上传的商品图片进行结构化识别。

**识别流程：**

1. 将预处理后的图片和精心设计的 Prompt 发送给 Qwen-VL-Max
2. Prompt 包含指令：识别商品的品类（category）、品牌（brand）、型号（model）、成色描述（condition）
3. Qwen-VL-Max 返回结构化 JSON 数据
4. 后端解析 JSON，提取四个字段展示给用户

**双通道容错机制：**

Qwen-VL 有两种调用方式：
- **OpenAI 兼容接口**：更标准，优先使用
- **DashScope 原生 SDK**：阿里云百炼专有接口，作为备用

系统实现自动容错：优先使用 OpenAI 兼容接口，失败后自动切换到 DashScope 原生 SDK。两个通道都失败才会向用户报错。

**YOLO 物品检测：**

YOLOv8 是轻量级本地检测模型，基于 COCO 数据集预训练，可识别 80 类常见物品。

**检测流程：**

1. 加载 yolov8n.pt 预训练权重（约 6.5MB）
2. 将图片输入模型，前向传播一次（仅需几十毫秒）
3. 输出检测到的物体列表：类别名称、置信度、边界框坐标
4. 在原图上用不同颜色画框标注
5. 输出带标注的 Base64 图片和检测结果列表

**双模型比对：**

为了提高准确率，系统让 YOLO 和 Qwen-VL 同时进行识别，然后比对两者的结果：

- **一致**：高置信度结果，直接展示给用户
- **不一致**：将差异信息自动记录到 `hard_cases` 错题本（包含图片 URL、YOLO 结果、Qwen-VL 结果），用于后续模型训练改进

这个设计体现了"数据飞轮"的核心思想：用云端大模型（Qwen-VL）的准确性来弥补本地小模型（YOLO）的不足，同时积累数据用于持续训练本地模型。

**识别结果展示：**

用户可以在前端查看并修正 AI 识别的结果：
- 品类：手机 / 笔记本 / 平板 / 耳机 / 手表 / 外设 / 其他
- 品牌：系统自动匹配已知品牌
- 型号：AI 识别的具体型号
- 成色：AI 对商品新旧程度的描述

### 4.4 瑕疵检测与分级

**7 类 Kaputt 缺陷分类：**

Kaputt 是德语"损坏"的意思，是一套工业级损伤分类标准。系统训练了专门的 YOLOv8-seg 分割模型来检测这 7 类缺陷：

| 缺陷类型 | 英文名 | 典型表现 | 检测依据 |
|---------|--------|---------|---------|
| 穿透 | puncture | 屏幕被戳穿、外壳被利器刺穿 | 深色孔洞、边界尖锐 |
| 变形 | deformation | 机身摔弯、外壳凹凸不平 | 轮廓不规则、反光异常 |
| 功能故障 | functional_fault | 按键失效、接口损坏 | 部件缺失或形变（需结合上下文） |
| 结构损坏 | structural_damage | 外壳开裂、铰链断裂 | 裂纹纹理、连续性中断 |
| 溢漏 | leakage | 屏幕漏液、电池漏液 | 不规则色块、液痕纹理 |
| 表面瑕疵 | surface_defect | 划痕、掉漆、氧化变色 | 细长纹理、色差 |
| 部件缺失 | missing_part | 少了按键、后盖、螺丝 | 规则缺失区域、边缘对比 |

**4 级严重程度自动分级：**

检测到瑕疵后，系统计算瑕疵的像素面积占整张图的面积比例，按比例自动分级：

| 级别 | Area Ratio | 标注样式 | 颜色 | 含义 | 对价格的影响 |
|------|-----------|---------|------|------|------------|
| 重度 | > 5% | 圆形框 | 红色 #FF0000 | 严重损伤，明显影响使用 | 大幅降价（扣 30%-50%） |
| 中度 | 1.5% ~ 5% | 矩形框 | 橙色 #FF8C00 | 中等损伤，影响美观 | 中等降价（扣 15%-30%） |
| 轻度 | 0.3% ~ 1.5% | 多边形框 | 金色 #FFD700 | 轻微损伤，仔细看才能发现 | 小幅降价（扣 5%-15%） |
| 轻微 | < 0.3% | 虚线框 | 蓝色 #00BFFF | 极细微损伤，几乎不影响 | 基本不降价 |

**分级策略的设计考量：**

严重程度分级**不直接展示给用户**。用户界面上只显示"检测到 X 个瑕疵"，而不显示具体分级标签。这是因为：
- 如果用户看到"重度损伤"四个字，可能直接放弃发布，减少了平台的商品供给
- 严重程度是内部传给 DeepSeek 做定价参考的参数，不需要向用户暴露
- 瑕疵标注图已经直观展示了缺陷的情况，用户能自行判断

**标注图渲染：**

1. 根据每个瑕疵的边界框坐标和严重等级，用对应样式在原图上画框
2. 使用跨平台字体工具（Windows 用微软雅黑、macOS 用苹方、Linux 用文泉驿）写中文标签
3. 生成标注后的图片保存到服务器，URL 返回给前端展示

**模型部署：**

- 瑕疵检测模型使用单例模式加载，整个进程只有一个检测器实例
- 模型权重加载需 3-5 秒，占用约 200MB 内存
- 首次加载后全局复用，避免每个请求都重复加载

**三层匹配策略：**

匹配从精确到模糊逐层降级，确保尽可能返回有效结果：

1. **精确匹配（第一层）**：brand + model 与数据库完全一致 → 直接返回。这是最理想的情况，比如 AI 识别到 "Apple iPhone 14"，数据库里正好有这条记录。

2. **模糊匹配（第二层）**：同品牌下，用 SequenceMatcher 算法比较型号字符串的相似度。相似度超过 70% 的视为匹配项。比如 AI 识别到 "iPhone 14 Pro"，数据库中只有 "iPhone 14"，通过模糊匹配可以关联上。

3. **未命中（第三层）**：以上两层都没有匹配结果时，返回空（None），前端提示用户手动输入价格。这是合理的降级处理，因为数据库中不可能覆盖所有二手商品的行情。

**查询流程：**

1. 用户在确认商品信息后，点击"查价"按钮
2. 后端用品牌 + 型号执行三层匹配查询
3. 返回均价、最低价、最高价
4. 前端以卡片形式展示行情数据，供用户参考

### 4.6 智能定价

智能定价是系统的核心创新功能之一，它综合瑕疵检测结果和市场行情，通过大语言模型推理给出有理有据的建议售价。

**定价流程：**

1. **数据准备**：
   - 瑕疵检测结果（缺陷类型、严重程度、面积占比）
   - 市场行情数据（均价、最低价、最高价）
   - 商品基础信息（品类、品牌、型号、成色）

2. **Prompt 构建**：
   系统构建详细的 Prompt，包含：
   - 商品的基本信息
   - 市场行情参考价格
   - 每个瑕疵的类型、严重程度和面积
   - 要求模型给出建议售价和扣款明细

3. **DeepSeek 推理**：
   将 Prompt 发送给 DeepSeek，模型基于市场行情和瑕疵情况推理出合理售价，并说明每一项扣款的理由。例如：
   - 市场均价：¥3,200
   - 屏幕划痕（中度）：-¥300
   - 边框磕碰（轻度）：-¥100
   - 电池健康度 85%：-¥200
   - 建议售价：¥2,600

4. **价格回退保护**：
   如果查价未命中（数据库中没有对应行情），DeepSeek 会给出一个默认估值。系统对此做了保护：
   - 当查价失败但 DeepSeek 给出价格时，添加 `price_fallback` 标记
   - 前端显示提示消息："⚠️ 未查到该商品行情，以下为 AI 估算价格，仅供参考"
   - 防止用户将无依据的估算价格当作市场行情

**设计考量：**

- 定价逻辑不硬编码：不同品类、不同缺陷对价格的影响差异很大，硬编码规则无法覆盖所有情况。使用 LLM 推理能适应各种复杂情况。
- 扣款明细可解释：用户不仅知道建议售价，还知道为什么要扣款，增强定价的可信度。
- 非强制约束：建议售价仅为参考，用户可以根据自己的判断进行调整。

### 4.7 文案生成

文案生成是用户感知最强的功能之一——通过 SSE（Server-Sent Events）流式技术，让 AI 生成的文案逐字出现在屏幕上，产生"打字机"效果。

**生成流程：**

1. **Prompt 构建**：
   - 商品信息：品类、品牌、型号、成色
   - 瑕疵情况：检测到的缺陷类型和程度
   - 建议售价
   - 生成要求：200 字精品文案，突出卖点，适合二手交易场景

2. **SSE 流式调用 DeepSeek**：
   - 后端调用 DeepSeek 的流式 API（stream=True）
   - DeepSeek 每生成一个 token（大约 1-2 个汉字）就通过回调返回
   - 后端立即将 token 包装成 SSE 格式推送给前端
   - SSE 数据格式：`data: {"type": "content", "text": "..."}\n\n`

3. **前端接收与渲染**：
   - 使用浏览器 fetch API 配合 ReadableStream 逐块读取
   - 每收到一个 token 立即追加到页面文本区域
   - 使用 requestAnimationFrame 做渲染节流（合并多个 token 一次性更新 DOM）
   - 支持用户随时点击"取消"按钮中断生成（通过 AbortController 实现）

**SSE 事件类型：**

| 事件类型 | 触发时机 | 数据内容 |
|---------|---------|---------|
| `start` | 开始生成 | `{"type": "start", "message": "开始生成文案..."}` |
| `content` | 每个 token | `{"type": "content", "text": "商品"}` |
| `done` | 生成完成 | `{"type": "done", "full_text": "完整文案"}` |
| `error` | 生成出错 | `{"type": "error", "message": "生成失败，请重试"}` |

**为什么选择 SSE 而不是 WebSocket？**

- 文案生成是单向的：客户端传参数 → 服务器推送文案，不需要双向通信
- SSE 基于标准 HTTP，不需要协议升级
- SSE 浏览器原生支持自动重连
- SSE 实现更简单，资源开销更小

**关键 HTTP 头配置：**

```
Cache-Control: no-cache
Connection: keep-alive
X-Accel-Buffering: no
```

- `Cache-Control: no-cache` 防止浏览器缓存流式内容
- `Connection: keep-alive` 保持长连接
- `X-Accel-Buffering: no` 告诉 Nginx 不要缓冲（生产环境部署时使用）

**渲染节流设计：**

每次 LLM 推送 token 的频率很高（每秒 10-50 个 token），如果每次都直接更新 DOM 会导致严重的性能问题。系统使用 `requestAnimationFrame` 做节流：
1. 收到 token 后暂存到缓冲区
2. 等待下一个 animation frame
3. 一次性将缓冲区中的所有 token 追加到 DOM
4. 视觉上仍然流畅（逐字出现），但实际 DOM 操作减少了 90% 以上

### 4.8 商品发布与管理

**保存与发布：**

用户在确认商品信息（品类、品牌、型号、成色、售价、文案）后，可以选择：
- **保存为草稿**：status = 'draft'，商品不进入商城
- **立即发布**：status = 'published'，商品进入商城可见

**数据保存流程：**

1. 收集所有商品信息：用户 ID、图片 URL、识别结果、瑕疵数据、AI 文案、建议售价
2. MySQL 事务保存：
   - 写入 `published_items` 表（核心业务表）
   - 更新 `hard_cases`（如果双模型比对不一致）
   - 记录 `ai_audit_logs`（AI 调用审计）
3. 如果状态为 published，异步执行 Qdrant 索引：
   - 下载商品图片
   - CLIP 提取 512 维特征向量
   - 存入 Qdrant（附带商品 ID、标题、品类、价格等元数据）

**状态流转：**

```
draft ──→ published ──→ delisted
  ↑                       │
  └───────────────────────┘ (重新发布)
```

| 状态 | 含义 | 商城可见 | 触发操作 |
|------|------|---------|---------|
| draft | 草稿（未发布） | ❌ | 用户选择"保存为草稿" |
| published | 已发布 | ✅ | 用户选择"发布" / 草稿发布 |
| delisted | 已下架 | ❌ | 用户选择"下架" |

**状态变更的副作用：**

- 发布（draft → published）：CLIP 提取特征 → 存入 Qdrant 索引
- 下架（published → delisted）：从 Qdrant 删除对应向量，保证搜索结果的有效性
- 重新发布（delisted → published）：重新提取特征 → 重新存入 Qdrant

**数据存储结构：**

`published_items` 表包含以下核心字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| original_image_url | VARCHAR(500) | 原始图片存储路径 |
| bg_removed_url | VARCHAR(500) | 去背景图片路径（可选） |
| annotated_url | VARCHAR(500) | 瑕疵标注图路径（可选） |
| defect_count | INT | 检测到的瑕疵数量 |
| defect_data | JSON | 瑕疵详细数据（类型、位置、面积、严重程度） |
| ai_generated_title | VARCHAR(200) | AI 生成的商品标题 |
| ai_generated_desc | TEXT | AI 生成的商品描述（长文本） |
| suggested_price | DECIMAL(10,2) | AI 建议售价 |
| category / brand / model | VARCHAR | 品类 / 品牌 / 型号 |
| condition | VARCHAR(200) | 成色描述 |
| status | VARCHAR(20) | 发布状态（draft/published/delisted） |
| review_status | VARCHAR(20) | 审核状态（normal/approved/rejected） |

### 4.10 以图搜图

以图搜图是本产品区别于传统二手交易平台的核心特色功能。

**搜索流程：**

1. **用户上传查询图片**：支持 JPG/PNG/WEBP，大小限制同上传功能
2. **文件安全检查**：MIME 验证 + PIL 解码验证
3. **CLIP 特征提取**：使用 CLIP ViT-B-32 图片编码器提取 512 维浮点数特征向量
4. **Qdrant 向量检索**：在 Qdrant 的 HNSW 索引中查找与查询向量最相似的 Top-K 个点
5. **批量查询商品详情**：用返回的商品 ID 列表，通过 MySQL `IN` 查询一次性获取所有商品详情（避免 N+1 问题）
6. **返回结果**：商品列表 + 相似度分数，按相似度从高到低排列

**CLIP 模型配置：**

- 架构：Vision Transformer (ViT-B/32)
- 预训练权重：LAION-2B（包含 20 亿图文对的海量数据集）
- 图片预处理：缩放到 224×224 → 中心裁剪 → 标准化
- 输出维度：512 维 float32 向量

**Qdrant 配置：**

- 索引算法：HNSW（Hierarchical Navigable Small World）
- 距离度量：余弦相似度（Cosine Similarity）
- Collection 名称：items
- 元数据：商品 ID、标题、品类、价格

**品类过滤：**

搜索时支持通过 Qdrant 的 filter 功能按品类过滤：
- 比如用户只想搜"手机"类商品，向量检索只在品类为"手机"的点中进行
- 大大提升搜索结果的相关性

**Top-K 保护：**

限制 `top_k` 参数最大值为 100，防止用户传入极大值导致系统资源耗尽。

### 4.11 以文搜图

除了上传图片搜索，系统还支持输入文字描述来搜索商品。

**搜索流程：**

1. 用户输入文字描述（如"黑色 iPhone，屏幕完好，95新"）
2. CLIP 文本编码器将文字转换为 512 维特征向量
3. 在 Qdrant 中搜索与文本向量最相似的图片向量
4. 返回匹配商品列表

**为什么文字也能搜图？**

CLIP 的核心特性是它将图片和文字映射到了**同一个语义空间**。在这个空间里：
- 图片 "iPhone 14" 的向量
- 文字 "一台黑色的 iPhone" 的向量
- 它们的距离很近（语义相似）

因此，输入文字描述就能找到视觉上和描述匹配的商品图片。

### 4.12 价格历史分析

**价格走势图：**

- 展示三条曲线：均价（avg）、最低价（low）、最高价（high）
- 默认展示最近 180 天的数据

**AI 价格预估：**

- 基于历史价格数据，由 `history_estimator` 服务估算当前合理价格
- 用于补充或验证市场查价的结果
- 尤其在行情数据缺失时提供替代参考

**涨跌分析：**

- 展示统计周期内的最高价、最低价、均价

- 帮助用户判断当前是"买方市场"还是"卖方市场"

### 8.3 安全指标

| 安全机制 | 实现方式 | 防护目标 |
|---------|---------|---------|
| JWT 认证 | HS256 签名，请求头携带 | 防止未授权访问 |
| 密码安全 | PBKDF2-SHA256, 16 字节随机盐, 100k 次迭代 | 防彩虹表、防暴力破解 |
| 速率限制 | 滑动窗口，按端点独立限流 | 防恶意高频请求 |
| SQL 注入防护 | 全参数化查询 + ORDER BY 白名单 | 防 SQL 注入 |
| 文件安全 | MIME 验证 + PIL 解码 + 路径遍历防护 | 防恶意文件上传 |
| 身份防伪造 | user_id 从 JWT 提取，忽略请求体 | 防身份冒充 |
| XSS 防护 | 安全响应头中间件 | X-Content-Type-Options |
| 点击劫持防护 | 安全响应头中间件 | X-Frame-Options: DENY |
| 审计追溯 | 每次 AI 调用记录到 ai_audit_logs | 问题追溯、成本核算 |
