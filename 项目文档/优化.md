## 二、改动一：自建模型逐步替代 Qwen

### 2.1 现状问题

此前完全依赖 Qwen（阿里云）进行图像识别，存在以下风险：
- **单一模型依赖**：Qwen 服务不可用时，整个识别功能瘫痪。
- **无法自主迭代**：模型优化受限于第三方，无法根据业务数据快速调整。
- **成本考量**：长期依赖商业 API 存在费用压力。

### 2.2 改造方案

#### 2.2.1 双模型并行架构

- 自行构建图像识别模型（基于开源模型如 ResNet、EfficientNet、ViT 等），与 Qwen **并行运行**。
- 先从公开数据集（如 ImageNet、OpenImages 等）训练一个基础版本，确保能跑通大致流程。
- 每次用户上传图片后，**本地模型结果与 Qwen 结果同时输出**，并进行比对。

#### 2.2.2 错误数据收集机制

当本地模型与 Qwen 结果不一致时，判定为「错误图片」，执行以下保存操作：

**（1）图片保存**
- 将错误图片保存到专门的**错误图片文件夹**。
- 命名规则：`{原图ID}_{时间戳}.jpg`

**（2）标签文件保存**
- 为每张错误图片生成一个对应的 **TXT 文件**（同名）。
- TXT 文件内容格式：
  ```
  图片名称：item_001.jpg
  错误分类（本地模型输出）：鼠标
  正确分类（Qwen 输出）：鼠标垫
  比对时间：2026-06-23 14:30:25
  用户ID：xxx
  商品ID：xxx
  ```

**文件结构示例**：
```
error_data/
├── images/
│   ├── item_001_20260623.jpg
│   ├── item_002_20260623.jpg
│   └── ...
├── labels/
│   ├── item_001_20260623.txt
│   ├── item_002_20260623.txt
│   └── ...
└── metadata/
    └── error_summary_2026W25.json   # 每周汇总统计
```

#### 2.2.3 每周迭代训练

**流程**：
1. 以**每周**为周期（如每周一凌晨），触发训练任务。
2. 读取本周收集的所有错误数据（图片 + 正确标签）。
3. 调用本地模型进行**微调训练（Fine-tuning）**。
4. 训练完成后，更新本地模型权重，供下一周线上推理使用。

**训练输入**：
- 错误图片（图像数据）
- 正确分类标签（来自 Qwen 的输出，作为 Ground Truth）

#### 2.2.4 过拟合防范（重点）

为防止模型在错误数据上过拟合，采取以下多维度措施：

| 防范措施 | 具体做法 |
|----------|----------|
| **数据混合** | 每次训练时，不仅使用本周错误数据，还从**历史错误数据池**中按比例（如 1:1）抽样混合 |
| **早停法（Early Stopping）** | 预留验证集，当验证集准确率连续 N 轮不提升时停止训练 |
| **数据增强** | 对错误图片进行旋转、裁剪、翻转等增强操作，扩充样本多样性 |
| **正则化** | 使用 Dropout、Weight Decay 等正则化手段 |
| **学习率调度** | 使用余弦退火或阶梯式降低学习率，避免震荡 |
| **K 折交叉验证** | 在训练时划分 K 份数据，轮流作为验证集 |

**训练集构成示例**：
```
本周错误数据：200 张
历史抽样数据：200 张（从过去 4 周中各抽 50 张）
验证集：50 张（从上述混合数据中划分）
─────────────────────────────
总计训练样本：400 张
```

#### 2.2.5 训练数据补充

**爬虫采集**：
- 编写爬虫脚本，从**二手交易平台**（如闲鱼、转转等）抓取公开数据。
- 采集内容包括：**商品图片 + 分类标签**。
- 采集频率：建议每日增量抓取。
- 采集到的数据经**人工或自动清洗**后，可作为训练集的补充来源。
- 爬虫数据与错误数据**分开存放**，在训练时可选择是否混合使用。

**注意合规性**：
- 遵守目标网站的 robots.txt 规则。
- 控制请求频率，避免对目标服务器造成压力。
- 仅采集公开可访问的数据，不涉及用户隐私。

### 2.3 数据流转

> 去完背景的「干净图片」→ 同时提供给 **Qwen** 和**本地训练的分类模型**使用。
>
> 两个模型的输出结果 → 比对模块 → 一致则放行，不一致则存入错误文件夹。


## 三、改动二：引入瑕疵检测与画圈标注

### 3.1 现状问题

之前的识别只输出物品名称，不涉及损耗/瑕疵信息：
- 用户无法了解商品的真实损耗情况。
- 二手商品的「折旧价值」在系统中几乎没有体现。
- 定价完全依赖人工经验，缺乏数据支撑。

### 3.2 改造方案

#### 3.2.1 瑕疵检测模型

- 引入目标检测模型（如 **YOLOv8 / YOLOv11**），在商品图上进行**画圈/画框标注**。
- 精确框选出瑕疵/损伤区域，支持的瑕疵类型包括但不限于：
  - 划痕
  - 磕碰/凹陷
  - 掉漆/褪色
  - 裂纹
  - 污渍
  - 变形

#### 3.2.2 不同形状与颜色代表不同程度

**标注规则**：不同**形状** + 不同**颜色** 组合代表不同的瑕疵程度和类型。

| 形状 | 颜色 | 代表含义 | 程度等级 |
|------|------|----------|----------|
| 圆形 | 红色 | 严重损伤（影响功能） | 重度 |
| 矩形 | 橙色 | 中度损伤（影响外观） | 中度 |
| 多边形 | 黄色 | 轻微损伤（细微痕迹） | 轻度 |
| 虚线框 | 蓝色 | 表面污渍/可清理 | 轻微 |

**（注：上述颜色与程度的对应关系，不向用户披露）**

#### 3.2.3 对外展示规则

- **画圈标注后的图片** → 仅**展示给用户**（前端显示画圈效果），让买家直观看到瑕疵位置和分布。
- **程度信息（颜色/形状的含义）** → **不告诉用户**，仅供内部参考和 DeepSeek 定价使用。
- 画圈后的图片**不保存到本地**，仅做临时展示用途（减少存储成本，且避免用户误解）。

#### 3.2.4 信息流转

- 瑕疵的**具体位置（坐标） + 数量（几处） + 程度等级** → 以结构化数据形式发送给 **DeepSeek**。
- DeepSeek 结合商品类别、市场行情、瑕疵严重程度，给出**参考价格调整建议**（如降价 5%、10% 等）。
- 最终定价由卖家确认，AI 建议仅作为辅助参考。

#### 3.2.5 输出格式

```
检测结果：
- 商品名称：iPhone 14 Pro
- 瑕疵总数：3 处
  - 瑕疵1：位置 (120, 340, 180, 400)，类型：划痕，程度：轻度
  - 瑕疵2：位置 (200, 150, 260, 220)，类型：磕碰，程度：中度
  - 瑕疵3：位置 (50, 480, 130, 520)，类型：掉漆，程度：轻度
→ 发送给 DeepSeek 进行定价评估
```


## 四、改动三：增加图像预处理流程

### 4.1 现状问题

用户上传的图片常包含以下干扰：
- 桌角、背景杂物等无关内容。
- 多物体同框（如鼠标 + 鼠标垫同时出现）。
- 图片噪点多、清晰度不一。
- 尺寸不统一，影响模型输入。

### 4.2 改造方案

#### 4.2.1 尺寸统一

- 用户上传图片后，首先进行**尺寸统一**。
- 所有图片调整为固定尺寸（如 **448×448** 或 **512×512**）。
- 后续所有处理（去噪、去背景、识别、特征提取）都基于此标准尺寸进行。

**尺寸选择考量**：
- 太小（如 224×224）：可能丢失细节，影响瑕疵检测精度。
- 太大（如 1024×1024）：增加计算负担，影响响应速度。
- 448×448 或 512×512 为平衡方案。

#### 4.2.2 去背景（rembg）

**当前状态**：待开发（尚未编写去背景代码）。

**技术方案**：
- 使用 `rembg` 库（基于 U²-Net）去除图片背景，分离商品主体。
- 处理流程：输入原始图片 → rembg 处理 → 输出透明背景的商品图。

**多物体处理策略**：
- 当画面中存在多个物体时（如鼠标和鼠标垫同时出现在画面中），进行**面积占比筛选**：
  - 使用轮廓检测或语义分割，识别各个物体。
  - 计算每个物体的像素占比。
  - 选择**占比最大（如 > 60%）** 的物体作为主要识别目标。
  - 其他物体视为背景干扰，一并去除。
  - 若占比接近（如 50% vs 50%），则提示用户重新上传单物品图片。

#### 4.2.3 去噪与图像增强

- 在去背景后进行**去噪处理**，清除图片中的杂点、纹理干扰。
- 常用去噪算法：高斯滤波、中值滤波、双边滤波、非局部均值去噪等。

**关键注意**：
- **避免过度去噪**：需参考行业通用方法（如 OpenCV 官方文档推荐的参数范围），平衡去噪强度与细节保留。
- 过度去噪的后果：商品表面纹理、标识文字、细微瑕疵等关键信息丢失，影响后续识别和瑕疵检测。
- 建议通过 A/B 测试确定最佳参数。

#### 4.2.4 预处理流程图

```
原始图片（用户上传）
       ↓
┌──────────────────┐
│ 尺寸统一         │ → 固定 448×448
└──────────────────┘
       ↓
┌──────────────────┐
│ 多物体判断       │ → 选占比最大的物体
└──────────────────┘
       ↓
┌──────────────────┐
│ 去背景（rembg）   │ → 透明背景商品图
└──────────────────┘
       ↓
┌──────────────────┐
│ 去噪             │ → 高斯滤波/双边滤波
└──────────────────┘
       ↓
┌──────────────────┐
│ 预处理完成       │ → 用于后续识别和特征提取
└──────────────────┘
```

### 4.3 原始图片保留

- 本地数据库**始终保存原始图片**（用户上传的未经任何处理的版本）。
- 预处理后的图片仅用于识别、检索等分析环节，不覆盖原始图。
- 原始图用途：
  - 数据备份与溯源。
  - 用户可随时查看原始上传内容。
  - 预处理参数调整后可重新处理，无需用户重新上传。


## 五、改动四：增加以图搜图能力

### 5.1 现状问题

此前商品匹配仅支持**文字搜索**：
- 用户难以用图片直接查找相似商品。
- 文字描述不准确时，搜索结果不理想。
- 无法实现「看到同款，快速找货」的场景。

### 5.2 改造方案

#### 5.2.1 特征向量提取

- 为每一张商品图片（预处理后的干净图片）提取**特征向量**。
- 特征提取模型：使用轻量级网络如 **MobileNetV3**、**EfficientNet-Lite** 或 **ResNet50**。
- 输出维度：通常为 512 维或 1024 维浮点数向量。
- 提取后的向量进行 **L2 归一化**，便于后续相似度计算。

#### 5.2.2 特征向量存储

- 所有已上架商店的商品图片特征向量，全部保存到**向量索引库**。
- 索引技术：**FAISS**（Facebook AI Similarity Search），支持 CPU 上毫秒级检索。
- 存储结构：
  ```
  FAISS 索引：
  ┌─────────┬─────────────────────────────────────────────┐
  │ item_id │ 特征向量（512 维 float32）                   │
  ├─────────┼─────────────────────────────────────────────┤
  │ 10001   │ [0.123, 0.456, 0.789, ...]                 │
  │ 10002   │ [0.234, 0.567, 0.890, ...]                 │
  │ ...     │ ...                                         │
  └─────────┴─────────────────────────────────────────────┘
  ```
- 特征向量与 MySQL 中的商品数据解耦，通过 `item_id` 关联。

#### 5.2.3 检索逻辑

当某用户上传一张新图片进行搜索时：

1. **提取特征**：对上传图片进行预处理 → 提取特征向量。
2. **全局比对**：使用 FAISS 与数据库中**全部已有商品的特征向量**进行比对。
3. **相似度计算**：计算余弦相似度或欧氏距离。
4. **返回结果**：按相似度从高到低排序，返回 Top-K（如 Top-10）最相似的商品列表。
5. **展示**：前端展示相似商品的图片、名称、价格等信息。

#### 5.2.4 索引更新时机

| 时机 | 操作 |
|------|------|
| 新商品上架 | 提取特征 → 加入 FAISS 索引 |
| 商品下架 | 从 FAISS 索引中移除（或标记为不可用） |
| 商品图片更新 | 重新提取特征 → 更新索引 |
| 系统启动 | 加载已有索引到内存 |

### 5.3 数据流转

| 图片类型 | 存储位置 | 用途 |
|----------|----------|------|
| 原始图片（用户上传） | 本地数据库（MySQL 文件存储/OSS） | 备份与溯源 |
| 预处理后图片（统一尺寸、去噪、去背景） | 临时处理/缓存 | 用于识别、特征提取 |
| 特征向量 | FAISS 索引库 | 以图搜图匹配 |

## 七、数据流转规则汇总

| 数据/图片 | 处理环节 | 去向 | 是否保存 | 保存格式 |
|-----------|----------|------|----------|----------|
| 原始图片 | 用户上传 | 本地数据库 | **是**（永久） | 原图格式 |
| 预处理后图片（干净图） | 去背景+去噪+尺寸统一 | Qwen + 本地识别模型 | 临时缓存 | 内存/临时文件 |
| 错误图片（模型差异） | 双模型比对不一致 | error_data/images/ | **是** | JPG |
| 错误标签（差异说明） | 双模型比对不一致 | error_data/labels/ | **是** | TXT（含错误分类+正确分类） |
| 画圈标注后的瑕疵图 | YOLO 检测后 | 用户（前端展示） | **否**（仅临时渲染） | 不保存 |
| 瑕疵位置 + 类型 + 程度 | YOLO 检测后 | DeepSeek | **是**（结构化数据） | JSON |
| 特征向量 | 特征提取后 | FAISS 索引库 | **是** | FAISS index |
| 爬虫采集数据 | 定时抓取 | 训练数据集 | **是** | 图片 + 分类标签 |

### 8.1 响应时间要求

| 环节 | 目标耗时 | 说明 |
|------|----------|------|
| 图片预处理（去背景+去噪） | < 3 秒 | rembg 在 CPU 上运行，GPU 可加速 |
| 双模型识别（本地+Qwen） | < 5 秒 | 两个模型并行调用，取较慢者 |
| 瑕疵检测（YOLO） | < 2 秒 | YOLOv8n 轻量版可达到实时 |
| 以图搜图（FAISS 检索） | < 100 毫秒 | 10 万级索引在 CPU 上毫秒级响应 |
| 总发布流程 | < 10 秒 | 包含以上所有环节 |

1. **开发环境建议**：Windows 下 `rembg` 和 `torch` 易遇编译问题，建议使用 **conda 环境** 管理依赖。
2. **模型加载策略**：所有模型在服务启动时预加载到内存，避免每次请求重复加载。
3. **错误处理**：任何 AI 服务（Qwen、DeepSeek）不可用时，本地模型应能独立完成基本识别。
4. **静默收集**：错误数据收集不影响主业务流程，采用异步方式处理。
5. **数据隐私**：爬虫采集遵守目标网站合规要求；用户数据不上传第三方。
6. **版本管理**：每周训练产出的模型权重按版本号（如 v2026W25）保存，支持回滚。


> **总结**：本系统围绕「双模型并行 → 收集错误 → 每周迭代」形成核心学习闭环；通过「预处理 → 瑕疵检测 → 以图搜图」完善商品分析能力；爬虫持续补充训练数据；所有数据流转规则清晰明确。最终实现一个**数据驱动、持续进化、自主可控**的二手商品智能识别与检索系统。
